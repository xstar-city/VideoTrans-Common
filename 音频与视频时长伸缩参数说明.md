# 音频时长伸缩参数说明

## 核心约束

视频画面、背景音轨**严格保持原始时长**，不做任何拉伸/调速。
每段 TTS 合成音频的时长必须贴合原始片段时长，由此自然得到：

```
combined.mp3 时长 = 原始音频时长
final.mp3 时长   = max(combined.mp3, others)
mux 后视频时长   = 原视频时长
```

任何"片段时长漂移 → 时间线错位"的问题都不会再出现。

## 三层兜底机制：把 TTS 段长强制贴到原段长

不允许"伸缩背景/视频来吸收差异"后，TTS 段长必须由 TTS 链路自己负责贴合，按以下顺序兜底：

| 层 | 手段 | 作用 |
|---|------|------|
| 第 1 层 | LLM 翻译反馈循环（仅 `tts_aware` 模式） | 翻译 → TTS 试合成 → 检查时长 → 超范围则 LLM 改写措辞重试，默认 3 次 |
| 第 2 层 | 模型重合成（仅支持 `duration` 参数的 TTS，如 Omni-Voice） | 把目标时长喂回模型，直接生成等长音频 |
| 第 3 层 | librosa 信号级时长拉伸 | 所有 TTS 的最终兜底，把音频物理拉到目标时长 |
| 第 4 层 | 截断保护 | 合成结果超过 `TTS_MAX_AUDIO_TRUNCATE_RATIO` 倍参考时长直接截断 |

## 拉伸幅度的内部限位

两个常量仍保留在 `服务端/config.py`，作为**第 3 层 librosa 拉伸的安全限位**和**第 1 层 LLM 反馈循环的"超范围"判定阈值**，**不再以 CLI 参数对外暴露**：

| 常量 | 默认值 | 含义 |
|------|--------|------|
| `TTS_MAX_AUDIO_SLOWDOWN_PCT` | 0.1 | 合成短于参考时，librosa 最多放慢 +10% |
| `TTS_MAX_AUDIO_SPEEDUP_PCT`  | 0.2 | 合成长于参考时，librosa 最多加快 -20% |

判定逻辑：原始片段时长 `orig` 秒，TTS 合成 `tts` 秒，可接受范围为

```
[orig × (1 - max_audio_speedup_pct), orig × (1 + max_audio_slowdown_pct)]
= [orig × 0.8, orig × 1.1]   （默认）
```

超出此范围 LLM 会被要求改写翻译；改写多次仍超出则由 librosa clamp 到边界（再大的差异也会被强行拉到 ±10%~20% 内）。

## 历史变化

旧版本（已下线）曾允许：
- 背景音轨按 TTS 时长拉伸（`AudioProcessing/audio_sync.py`）
- 视频画面按 TTS 时长拉伸（`Common/video_utils/sync.py`）
- 翻译阶段允许范围 = 音频范围 + 视频范围（合计余量）

这套机制依赖"累积时间漂移"算法，每段 TTS 比原段长/短的差额会累加到后续片段的起点。
当前置时长拉伸不彻底时会产生明显的"音视频错位"，且 final.mp3 末尾被裁切。

新版本彻底删除了背景/视频的伸缩链路，把时长贴合的全部责任收敛到 TTS 段自身：
- `combined.mp3` 直接按文件名表达的起点落位，不做任何累积偏移
- `final.mp3` 用 `duration="longest"` 与 others 等长叠加，不再 sync
- 视频画面用原视频直接 mux，不再切分调速合并

如有需要可在 git 历史中查看旧实现。
